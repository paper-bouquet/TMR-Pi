# TMR-Pi：高可靠度分散式容錯系統

## 專案簡介 (Project Overview)

在現實世界中，硬體隨時可能因為老化、過熱或外力干擾而產生「運算錯誤」，網路也可能隨時斷線。如果這台電腦正在控制一架飛機，一個位元的錯誤都可能導致災難。

**TMR-Pi (Triple Modular Redundancy on Raspberry Pi)** 就是為了解決這個問題而誕生的。

本專案將三台樹莓派 (Raspberry Pi) 綁定成一個生命共同體。當系統接收到任務時，三台機器會「同時」進行運算，並利用軍規級的 AES 加密網路互相對答案。
簡單來說：只要有兩台機器算出一樣的答案，系統就會採信；如果有機器當機或被駭客竄改資料，系統會自動將其剔除，確保最終輸出的結果 100% 正確。

---

## 核心亮點 (Core Value for Proposals)

本系統的四大防護機制：

1. **拜占庭容錯 (Majority Voting)**：不怕單一節點硬體故障或算出錯的答案，系統會自動少數服從多數 (3TMR 多數決)。
2. **網路孤島與降級機制 (Fault Degradation)**：當一台機器網路斷線時，系統不會癱瘓，而是自動降級為「兩人共識決 (2MR)」，確保任務持續運行。
3. **主動式心跳監測 (Heartbeat Detection)**：系統每 5 秒互相發送一次安全信號，超過 15 秒沒回應即判定死亡，隨時掌握叢集健康度。
4. **防篡改資安防護 (HMAC-SHA256)**：所有溝通封包皆帶有密碼學驗證碼。若駭客從中攔截並竄改選票，系統會立刻識破並銷毀該張假選票。

**適合的應用情境：**

* **神經工程與醫療儀器**：如 BCI 腦波特徵解碼，確保傳遞給機械義肢的控制訊號零誤差。
* **自駕車與航太控制**：在高速移動且高輻射干擾的環境中，維持核心運算的絕對正確。
* **金融交易節點**：防止單一伺服器遭駭客入侵導致的假帳本同步。

## 軟硬體規格 (System Specifications)

本系統經過嚴格的編譯與執行期測試，具備跨平台與邊緣運算部署能力。

**硬體環境 (Hardware)**

* **運算節點**：3 台 Raspberry Pi (或其他 Linux 主機)，分別命名為 Node A, B, C。
* **網路拓樸**：固定區域網路 (LAN)，使用 UDP Port `8888` 進行多播通訊。
* Node A IP: `192.168.50.41`
* Node B IP: `192.168.50.14`
* Node C IP: `192.168.50.62`



**軟體與技術棧 (Software Stack)**

* **程式語言**：C++17 (具備現代化多執行緒與記憶體安全特性) 


* **建置工具**：CMake (最低需求版本 v3.20.0) 


* **密碼學核心**：OpenSSL v3.5.5 (提供 AES-256-CBC 加密與 HKDF 金鑰衍生) 


* **並發控制**：POSIX Threads (`std::thread`, `std::mutex`, `std::condition_variable`)

## 測試與展示指南 (Demo Guide)

### Step 1: 環境初始化

在三台樹莓派上編譯好程式後，**必須先設定同一把大門鑰匙**，系統才能啟動。請在三台機器的終端機都輸入：

```bash
export AES_MASTER_KEY="0123456789abcdef0123456789abcdef0123456789abcdef0123456789abcdef"

```

### Step 2: 啟動叢集

分別在三台樹莓派上執行對應的身分：

* 第一台：`./build/TMR-Pi A`
* 第二台：`./build/TMR-Pi B`
* 第三台：`./build/TMR-Pi C`

### Step 3: 情境展示指令操作

在終端機的 `>` 提示符號後輸入以下指令來測試：

#### 展示 A：確認叢集連線 (狀態監控)

* **輸入**：`status`
* **看點**：畫面上會印出三台機器的連線狀況。如果剛開機，會看到大家從 `[未知]` 變成 `[在線]` 。

#### 展示 B：完美容錯 (模擬硬體故障)

* **操作**：在 B 機器上輸入 `fault` (開啟模擬錯誤)。接著在 A 機器上輸入任意英數字 (例如 `test_signal`) 發送任務。
* **看點**：A 機器會收到三份答案，但它會立刻發現 B 算錯了！不過因為 A 和 C 算得一樣，系統會顯示 **`[成功] 3TMR 多數決成功 (2/3)`**。這證明了「一台壞掉不影響大局」。

#### 展示 C：模擬網路斷線

* **操作**：直接拔掉 C 機器的網路線 (或強制關閉 C 的程式)。等待 15 秒後，在 A 輸入 `status`，會看到 C 顯示 **`[斷線]`**。
* **看點**：此時在 A 再次發送任務，系統會等 3 秒鐘確認 C 真的沒救了，接著印出 **`[降級成功] 2MR 一致`**。這證明了系統懂得在失去隊友時，自動切換成兩人相依為命的模式繼續工作。

#### 展示 D：網路孤島

* **操作**：把 B 和 C 的程式都關掉，只留 A。在 A 發送任務。
* **看點**：A 會發現自己被孤立了 (缺失 2 台)，於是觸發最終防護機制，印出 **`[嚴重失敗] 網路孤島狀態`** 並拒絕輸出任何不可靠的結果。
